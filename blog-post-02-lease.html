SECTION 01 — The Actual Problem

Real estate operations and legal teams often try to implement RAG to answer queries directly from lease and contract documents. It almost always fails in production. Building a prototype on a few PDFs is easy, but making it reliable for an entire residential or commercial portfolio exposes three fatal engineering flaws.

FAILURE MODE 01

Unstructured Data Has No Query Interface

A lease PDF is unstructured. Keyword search misses clauses because the user searches 'no pets' but the lease says 'animals and domestic creatures are prohibited.' Vector search solves this because it works on meaning, not exact characters.

FAILURE MODE 02

Structured Metadata Mixed Into Unstructured

A lease contains both types. 'Which leases expire in June 2026' should never touch a vector database — that is a SQL query. Routing this through RAG is slower, more expensive, and less accurate.

FAILURE MODE 03

The System Goes Stale After Day One

If you index 500 leases and deploy — it works on day one. On day 60, new leases arrive and old ones expire. Nothing in a static RAG detects this. The solution is an event-driven update pipeline, not a smarter prompt.

SECTION 03 — Correct Architecture: Three Layers

To safely deploy contract intelligence into production, we isolate the architecture into three strict layers working in tandem.

LAYER 01 — STATIC DOCUMENT LAYER (RAG)

PDF → OCR → Chunker → Embedder → Qdrant

↓
LAYER 02 — LIVE STRUCTURED LAYER (DATABASE)

Supabase · leases · amendments · alerts

↓
LAYER 03 — AUTOMATION LAYER (EVENT BRAIN)

n8n · Trigger · Router · Notifier · Logger

Layer 1 (Qdrant) handles pure unstructured text meaning. Layer 2 (Supabase) tracks absolute facts like dates and amounts. Layer 3 (n8n) acts as the central nervous system, ensuring any document upload automatically syncs to both databases.

SECTION 04 — Data Flow End to End

Upload Flow

1
Upload: Property manager uploads lease PDF to storage
2
Webhook: Upload triggers n8n webhook automatically
3
Parse: n8n calls Document Parser (OCR & SHA-256 hash)
4
Extract: Metadata Extractor (LLM) writes SQL to Supabase
5
Chunking: Split into 512 tokens, 64 overlap, tagged with lease_id
6
Vector Embed: Batch embed + store inside Qdrant
7
Status: Update Supabase index_status = 'indexed'

SECTION 05 — Update Strategy

Scenario Detection Method Action Cost Level
New lease uploaded Storage webhook Full ingestion pipeline High
Lease amended Hash comparison Re-embed changed chunks Low-Medium
Status change Supabase webhook Update payload only (no re-embed) Very Low
Expiry approaching Daily n8n schedule WhatsApp + email alert Zero
# When amendment arrives: 1. Parse new document → compute SHA-256 hash 2. Query Supabase for stored_hash 3. If hash matches → STOP (no change) 4. If hash differs: a. Delete ALL vectors for lease_id from Qdrant b. Re-embed full document c. Update Supabase: hash, version += 1 d. Log in amendments table

SECTION 06 — Supabase Schema

[leases]
CREATE TABLE leases ( id UUID PRIMARY KEY, property_id UUID, tenant_name TEXT, start_date DATE, expiry_date DATE, doc_hash TEXT, index_status TEXT DEFAULT 'pending' );
[amendments]
CREATE TABLE amendments ( id UUID PRIMARY KEY, lease_id UUID REFERENCES leases(id), amendment_date DATE, changes_summary TEXT );
[lease_alerts]
CREATE TABLE lease_alerts ( lease_id UUID, trigger_date DATE, alert_type TEXT, notified BOOLEAN DEFAULT false );
